## [high/L] Cards are written after judging and never validated as shipped; card cannot serve as a repair action
- **subsystem:** cards-sequence · **kind:** missing-feature
- **problem:** The judge supports a context_card input (validate.py:74-75,84) but the pipeline never uses it: validate_candidate passes no card (validate.py:105-108) because cards are generated only after accept/reject and boundary snap (assemble/__init__.py:110-112). Consequences: (1) the shipped clip+card combination is never judged — a duplicative, truncated, or subtly wrong card goes out unchecked; (2) the repair loop's only remedy is growing the clip (expand_candidate, validate.py:130-242) even when a one-sentence card covering a referential unit would fix a missing_prerequisite/unresolved_reference more cheaply — clips get longer than necessary or are rejected (validate.py:339-343) when a card could have saved them; (3) eval judges WITH the card (metrics.py:37-38), so headline comprehension is measured on a different artifact than the pipeline gated on, systematically flattering results and masking card regressions.
- **evidence:** /Users/vincentfeng/Documents/practice/clips/backend/pipeline/assemble/validate.py:74-75,84,105-108,130-242,339-343; /Users/vincentfeng/Documents/practice/clips/backend/pipeline/assemble/__init__.py:110-112; /Users/vincentfeng/Documents/practice/clips/backend/eval/metrics.py:37-38
- **proposal:** Add a card-repair branch to validate_and_repair: when the failing gates are only prerequisites/references and the missing concept maps to a referential unit, generate the card early (reusing generate_context_card) and re-judge with context_card set — accept 'card-completed' clips without growing them; and add one final judge pass on the accepted clip+card so what ships is what was validated (also aligns pipeline gating with the eval measurement).

## [high/L] No per-clip re-judge or cheap reassembly, even though the cached Structure makes both nearly free — and quality-floor drops are silent and unrecoverable
- **subsystem:** edit-ux · **kind:** missing-feature
- **problem:** Two 'wrong clip kept' scenarios have no remedy: (a) after a manual boundary nudge or a doubtful verdict there is no way to re-run judge/repair on ONE clip; (b) near-miss candidates are silently discarded by the quality floor and max_clips truncation (assemble/__init__.py:101-104) and by dedupe merging (refine.py:154-183), so the user never sees that a better alternative existed and cannot 'undrop' it. Yet the expensive understanding pass is cached per video (orchestrator.py:157, models.py:191-207) and assembly is the cheap topic-scoped tail (72→100% of the progress bar), with the judge cached by frozenset(unit_ids) (validate.py:317) — so re-running assembly with different knobs (quality_floor, max_anchors, closure_max_span_s — all supported settings at config.py:133-135 but absent from the frontend Settings type, types.ts:34-38) costs seconds, not the full pipeline.
- **evidence:** backend/pipeline/assemble/__init__.py:101-104 (drop_weak + truncation, dropped specs discarded); backend/orchestrator.py:157 (load_structure cache hit skips understanding); backend/pipeline/understand/models.py:191-207 (persisted structure); backend/pipeline/assemble/validate.py:288-343 (per-candidate validate_and_repair, callable in isolation); backend/config.py:133-135 vs frontend/src/types.ts:34-38 (assembly knobs not exposed)
- **proposal:** Two endpoints on top of the cached structure: POST /jobs/{job_id}/clips/{n}/revalidate re-runs validate_and_repair on that clip's unit set (returns updated verdict + possibly expanded boundaries); POST /jobs/{job_id}/reassemble {quality_floor?, max_anchors?, max_clips?} re-runs only assemble_clips→sequence on the cached Structure and swaps job.clips. Additionally return dropped-by-floor specs (title, start/end, final_quality) in the snapshot as 'alternates' the user can promote — the drop/undrop half of editing.

## [high/M] Post-judge boundary mutations never re-validated; clip metadata (unit_ids, title, card) goes stale
- **subsystem:** boundaries · **kind:** robustness-gap
- **problem:** The judge approves the exact text of sentences i_start..i_end (validate.py:107 via _clip_text). After approval, the cut is mutated in ways that change content with no re-judge, no warning, and no metadata update: (1) min-duration extension appends unseen sentences (refine.py:75-81); (2) _dedupe same-facet merge unions two judged clips but keeps only the higher-scored clip's role/title/unit_ids/referential — full-path candidates with the same role always share a facet (adapters/base.py:173-175), so overlapping same-role clips silently merge into a span nobody judged, and the context card (generated later from that one candidate's referential, assemble/__init__.py:111-112) describes half the footage; (3) _trim_start_after moves the start past the previous clip's end, potentially cutting off the inlined context-closure sentences that made the clip pass the judge, updating sentence_start_idx but not unit_ids and appending no warning (refine.py:140-151,177-182); (4) boundary_adapt keeps stale pre-snap i_start/i_end alongside new start/end (boundary_adapt.py:45). boundary_score can't penalize any of this because none of these paths append warnings.
- **evidence:** /Users/vincentfeng/Documents/practice/clips/backend/pipeline/assemble/validate.py:70-71,107; refine.py:75-81,126-137,140-151,174-182; boundary_adapt.py:41-46; assemble/__init__.py:93-112; adapters/base.py:173-175; types.py:16 (unit_ids contract)
- **proposal:** After snap/dedupe, recompute the sentence span and derive the final clip text; if it differs from the judged text beyond a token threshold, either re-run judge_clip on the new text or append explicit warnings ('extended_for_min_duration', 'merged_with_<cand_id>', 'trimmed_start') that boundary_score penalizes. On merge, union unit_ids/referential from both candidates; on trim, drop unit_ids whose sentence_range falls before the new start and re-check that the anchor unit survives (drop the clip if not).

## [high/M] Degraded-punctuation fallbacks fabricate sentence terminators, making mid-sentence cuts invisible to boundary_score
- **subsystem:** boundaries · **kind:** robustness-gap
- **problem:** When punctuation restoration degrades, sentences_from_chunks sets ends_with_period=True for every caption chunk ('chunk boundary is an acceptable cut point') and window_sentences sets terminator='.' on arbitrary 12-second windows. _snap_one's is_valid_end then treats every caption edge as a period, so clips can start and end mid-sentence with zero warnings and boundary_score=1.0 — the quality filter and the 0.20 boundary weight in final_quality are blind to the worst boundary regime. Sentence-level warnings that DO exist ('chunk_boundary', 'windowed', 'degraded', 'unmatched'/interpolated timings from _interpolate) are never propagated into the clip's warnings tuple by _snap_one, so downstream scoring/repair can't react. Interpolated sentences (start=prev_end, end=next_start) can even yield mid-word cut times with align_confidence 0.
- **evidence:** /Users/vincentfeng/Documents/practice/clips/backend/pipeline/sentences.py:392 (ends_with_period=True for chunks), sentences.py:331-336 (windowed terminator='.'), sentences.py:250-261,287-293 (_interpolate + 'unmatched'), punctuation/service.py:263-268 (chunk fallback trigger), refine.py:61-71,106-116 (warnings built only from period search; Sentence.warnings ignored), scoring.py:27-33 (boundary_score sees only those warnings)
- **proposal:** In _snap_one, union the boundary sentences' own warnings into the clip warnings (e.g. 'start_on_chunk_boundary', 'end_windowed', 'end_interpolated') and add corresponding penalties in boundary_score. When the sentence index is chunk/window-based, force the precise Whisper pass (it is already gated on source=='supadata', which is exactly when the chunk fallback fires) or lower quality_floor expectations explicitly.

## [high/M] Known-needed card silently empty: blanket exception swallow, grounding filter can zero the card, and closure's `truncated` flag never reaches the spec
- **subsystem:** cards-sequence · **kind:** robustness-gap
- **problem:** generate_context_card returns "" on any LLM exception (context_card.py:53-56) and the grounding filter can drop every sentence (context_card.py:57-59) — with no retry, no fallback, and no marker. Yet closure explicitly flags clips whose REQUIRED contract elements or strong prereqs were pushed to referential (`truncated=True`, closure.py:110-111,139-141; types.py:24). That flag is dropped at the Candidate→spec conversion (candidate_to_boundary_input, boundary_adapt.py:17-29 has no 'truncated' key), so downstream nothing knows the clip ships with known-missing context and an empty card. This is exactly the 'empty card when needed' failure: silent, unfixable, invisible to the quality score and the UI.
- **evidence:** /Users/vincentfeng/Documents/practice/clips/backend/pipeline/assemble/context_card.py:53-59; /Users/vincentfeng/Documents/practice/clips/backend/pipeline/assemble/closure.py:110-111,139-141; /Users/vincentfeng/Documents/practice/clips/backend/pipeline/assemble/types.py:24; /Users/vincentfeng/Documents/practice/clips/backend/pipeline/assemble/boundary_adapt.py:17-29
- **proposal:** (1) Carry `truncated` through candidate_to_boundary_input into the spec. (2) When the generated card is empty but referential units exist (especially truncated clips), fall back to an extractive card built verbatim from the top referential unit's `summary` — grounded by construction, so the no-hallucination guarantee holds. (3) Append a 'missing_context_card' warning to spec['warnings'] so boundary_score/final_quality and the UI reflect it.

## [high/M] unit_ids diverge from the actual clip span: intermediate units are invisible to contract check, grounding, and repair; referential units can lie INSIDE the clip
- **subsystem:** closure · **kind:** robustness-gap
- **problem:** When closure inlines a far unit it grows the span but adds only that unit's id (closure.py:130-137); build_candidate then takes min/max sentence_range over inline units (candidates.py:91-92), so every unit between anchor and the far unit is physically in the clip yet absent from unit_ids. Consequences: (1) check_contract computes roles_present from unit_ids only (contracts.py:25), reporting required elements 'missing' that are already inside the clip, triggering pointless _fill_contract passes; (2) grounding_score averages source_confidence over unit_ids only (scoring.py:19-24); (3) expand_candidate's pull_graph walks needs of unit_ids only (validate.py:149-152), so dependencies of in-clip units are never repaired; (4) a graph prerequisite whose gap-to-span is 0 (it sits between two inlined units) can still be pushed to referential when max_extra_units is exhausted (closure.py:131-139), and the context card then prefaces the clip with 'earlier in the video...' text that is literally inside the clip (context_card.py:44), and eval treats it as outside (eval/metrics.py:87).
- **evidence:** closure.py:130-139; candidates.py:91-92; contracts.py:25; scoring.py:19-24; validate.py:149-152,161; context_card.py:44
- **proposal:** After closure/reindex, absorb every unit whose sentence_range falls within [i_start, i_end] into unit_ids (a one-pass sweep over units), and drop referential entries whose unit lies inside the final span. This makes unit_ids the true clip contents for contract check, grounding, repair targeting, sequencing prerequisites, and context cards.

## [high/M] Same-facet overlap merge produces a never-judged union clip carrying one candidate's verdict, unit_ids, and referential
- **subsystem:** closure · **kind:** robustness-gap
- **problem:** snap_candidates runs refine._dedupe over validated specs (boundary_adapt.py:46). For same-facet overlapping clips, _dedupe calls _merge (refine.py:174-175), which unions start/end/sentence indices but keeps ALL other keys from whichever candidate scored higher (refine.py:126-137) — so the merged clip's transcript span was never judged, while its completeness_score/verdict, unit_ids, anchor_id, and referential describe only half the content. Downstream, the context card is generated from that stale referential/anchor (assemble/__init__.py:112, context_card.py:44-48), grounding in eval re-verifies against the wrong id set (eval/metrics.py:87), and the stale i_start/i_end from candidate_to_boundary_input coexist with the merged sentence_start/end_idx. Separately, _snap_one returning None silently drops a fully-validated clip with no note (boundary_adapt.py:42-44).
- **evidence:** boundary_adapt.py:17-29,42-46; refine.py:126-137,174-175; pipeline/assemble/__init__.py:112; context_card.py:44-48
- **proposal:** On merge, union unit_ids and referential (minus referential now inside the union span), invalidate or re-run the judge on the merged span (the frozenset(unit_ids) verdict cache makes a re-judge cheap and correct), and recompute i_start/i_end. Log/collect a note when _snap_one drops a validated candidate so the run report explains missing clips.

## [high/M] Batch cutting is all-or-nothing and cut failures are silently swallowed (user asked for mp4s, silently gets embeds)
- **subsystem:** cut-export · **kind:** robustness-gap
- **problem:** cut_clips uses asyncio.gather without return_exceptions: the first failing clip raises immediately, discards the N-1 already-successful results, never writes clips.json, and leaves the sibling ffmpeg tasks running unawaited (orphaned processes still writing into the output dir). The orchestrator then does `except Exception: pass` — no log, no job.notes, no per-clip error — so a job with output_mode=cut finishes "done" with every path=None and the user has no idea cutting failed or why. The pre-cut download failure path is equally silent (video_path=None → skip cutting).
- **evidence:** backend/pipeline/cut.py:97 (gather, no return_exceptions), cut.py:81-83 (raise aborts batch); backend/orchestrator.py:216-225 (try/except pass, no logging), orchestrator.py:193-198 (download failure → silent embed fallback); no logger exists in orchestrator.py or cut.py at all
- **proposal:** gather(..., return_exceptions=True); keep successful cuts, attach {"cut_error": msg} to failed clips, write clips.json with whatever succeeded, and append a warning to job.notes (e.g. "2/8 clips could not be rendered; playing as embeds"). Add logging with the ffmpeg stderr tail.

## [high/M] No way to fix a single wrong clip — no boundary-adjust, no single-clip re-judge/re-cut
- **subsystem:** cut-export · **kind:** missing-feature
- **problem:** When one clip is mis-bounded (starts mid-thought, ends a sentence late) the user has zero recourse: the API surface is create-job / stream / get / zip / export only. Export re-cuts at the stored start/end, and its exists()-cache means even a hypothetical retry returns the same file. The only option is re-running the whole pipeline, which (known) is nondeterministic and changes the whole clip set. Export also mutates clip["path"] in memory only — never persisted to clips.json — so even that state is lost on restart.
- **evidence:** backend/main.py:64-164 (full route list; no PATCH/regenerate route); main.py:125-144 (export locked to stored start/end), export.py:71-72 (cache blocks re-render); backend/main.py:143 (in-memory path mutation only)
- **proposal:** Add PATCH /jobs/{job_id}/clips/{n} accepting start/end (or ±delta): re-snap to sentence boundaries via the existing _snap_one/refine machinery, rebuild embed_url, invalidate/delete the cached mp4, persist the updated clip to job.clips and clips.json. Optionally add POST /jobs/{id}/clips/{n}/repair to re-run the clip-only judge+repair loop on just that clip.

## [high/M] No clip mutation API — a finished clip cannot be nudged, re-snapped, dropped, or edited in any way
- **subsystem:** edit-ux · **kind:** missing-feature
- **problem:** The complete API surface after a job finishes is read-only: GET /jobs/{id}, GET zip, POST export, GET clip file (main.py:43-164 route list). If a boundary is slightly off, a clip is redundant, or a context card is wrong, the ONLY recourse is re-running the entire job — which re-runs anchor selection and LLM judging, so the user gets a different (noisy) clip set, not a fixed version of the clip they had. All the machinery for a cheap deterministic fix already exists: _snap_one accepts raw start/end times and derives sentence indices itself (backend/pipeline/refine.py:45-58, via _sentence_containing_time refine.py:34-42), sentences are reconstructable without LLM calls from the punctuation cache (backend/pipeline/punctuation/cache.py:1-11), and embed URLs are centrally generated (orchestrator.py:38-42).
- **evidence:** backend/main.py:64-164 (no PATCH/DELETE/edit route); backend/pipeline/refine.py:45-58,103-116 (_snap_one reusable for re-snap from arbitrary times); backend/pipeline/punctuation/cache.py:1-11 (sentence index rebuildable from disk); backend/orchestrator.py:38-42 (_embed_url regeneration point)
- **proposal:** Add PATCH /jobs/{job_id}/clips/{n} accepting {start_delta_s?, end_delta_s?, context_card?, dropped?}. Server rebuilds sentences from the punctuation cache, applies deltas to start/end, re-runs _snap_one (preserving min/max duration + period-end invariants), regenerates embed_url and duration, invalidates clip['path'], and returns the updated clip dict. Add a 'dropped: bool' flag honored by zip/export. UI: ±1s/±5s nudge buttons and a drop toggle on ClipCard; the re-snap guarantees the nudged clip stays well-formed.

## [high/M] Jobs are in-memory only: after a server restart, finished clips become unexportable and results unrecoverable (embed mode persists nothing)

Add GET /jobs listing persisted snapshots and a small 'recent results' list on the input screen. This is the substrate every edit endpoint needs anyway.

## [high/S] Precise-boundary pass failure discards judged clips or fails the whole job
- **subsystem:** boundaries · **kind:** robustness-gap

## [high/S] prerequisite_clips hints fire even when the clip defines the concept itself
- **subsystem:** cards-sequence · **kind:** robustness-gap
- **problem:** Graph.defines_needed_for (graph.py:29-42) returns True if ANY consumer unit requires a concept an earlier clip introduces — it never subtracts the concepts the consumer clip introduces internally. Closure/repair deliberately inline definer units to make clips self-contained, so a clip that contains its own definition of X still gets a 'watch first' hint pointing at every earlier kept clip that also introduces X (attach_prerequisites, sequence.py:13-18). On topic-focused runs where most clips share a concept chain, this yields dense, spurious prerequisite hints that contradict the self-containment guarantee the judge just enforced.
- **evidence:** /Users/vincentfeng/Documents/practice/clips/backend/pipeline/assemble/graph.py:29-42; /Users/vincentfeng/Documents/practice/clips/backend/pipeline/assemble/sequence.py:13-18
- **proposal:** Compute unmet = union(concepts_required of s.units) - union(concepts_introduced of s.units) per clip, and hint only earlier clips whose concepts_introduced intersect unmet (either fix defines_needed_for to take the consumer's own introductions into account, or do the subtraction in attach_prerequisites).

## [high/S] Relevance-LLM failure silently disables the topic filter: clips presented as 'about TOPIC' are selected topic-blind
- **subsystem:** closure · **kind:** robustness-gap
- **problem:** score_topic_relevance swallows any llm_json exception per 120-unit batch and defaults every unit in the batch to 0.5 (candidates.py:56-61). Since ANCHOR_REL_FLOOR is 0.25 (config.py:210), a total LLM outage means every anchor-role unit clears the floor and anchors are ranked purely by role priority — yet assemble's notes still report 'N clip(s) about "topic"' (__init__.py:118) with no degraded signal anywhere. Partial failures are worse: real scores from surviving batches mix with 0.5 defaults on the same scale, distorting both the floor test and the ranking multiplier (0.4 + 0.6*rel, candidates.py:79). There is no retry and no way for the user to distinguish 'LLM failed' from genuine relevance.
- **evidence:** candidates.py:56-61,79; config.py:210 (ANCHOR_REL_FLOOR=0.25); pipeline/assemble/__init__.py:53,118
- **proposal:** Retry a failed batch once; if it still fails, record a degraded marker (return it alongside the scores or append to a notes list) and surface it in assemble's notes ('topic filtering degraded — showing best generic moments'). Optionally treat all-default relevance as 'no topic filter' explicitly instead of pretending scores exist.

## [high/S] Failed ffmpeg cut leaves a partial .mp4 that is forever treated as a valid cached clip
- **subsystem:** cut-export · **kind:** robustness-gap
- **problem:** ffmpeg writes directly to the final output path. On failure (rc != 0) both cut and export raise without deleting the partially-written file. export_clip's cache check is just `if out.exists(): return {"path": served}`, so the next export of that clip returns the truncated/corrupt mp4 permanently; the zip endpoint and /clips/{video_id}/{file} also serve it. A crash/restart mid-cut produces the same poisoned cache.
- **evidence:** backend/pipeline/cut.py:61,81-83 (writes to final path, raises without unlink); backend/pipeline/export.py:71-72 (exists() == cache hit), export.py:83-85 (raises on rc!=0 but leaves partial file); backend/main.py:109-112 (zip adds any existing file), main.py:155-164 (serves any file in output dir)
- **proposal:** Write to a temp name (e.g. .tmp suffix) and atomically rename only after rc == 0 and an ffprobe duration check; unlink the temp on failure. Optionally validate cached files (nonzero size + probed duration within ~0.5s of end-start) before returning a cache hit.
